# Деплой HH-бота (облако)

Кратко: ASGI-приложение (FastAPI) + Postgres. Браузеры Playwright работают на
сервере в фоне (headless), по одному контексту на пару `(пользователь, сайт)`.

## Переменные окружения

См. `.env.example`. Обязательные в проде:

| Переменная | Назначение |
|------------|------------|
| `DATABASE_URL` | `postgresql+psycopg://user:pass@host:5432/db` |
| `APP_JWT_SECRET` | секрет подписи JWT (иначе сгенерится локальный `.jwt_secret`) |
| `APP_ENCRYPTION_KEY` | мастер-ключ Fernet для кред/прокси (иначе локальный `.enc_key`) |

В облаке `APP_*` берите из секрет-менеджера/KMS, не из файлов.

## Запуск через docker-compose

```bash
cp .env.example .env      # заполните APP_JWT_SECRET и APP_ENCRYPTION_KEY
docker compose up --build
```

Поднимется Postgres и приложение на `:8000`. При старте контейнер выполняет
`alembic upgrade head` (миграции), затем `gunicorn` (см. `gunicorn_conf.py`).
Профили браузеров и БД сохраняются на томах (`profiles`, `pgdata`).

## Миграции БД

Схемой в проде управляет Alembic (не `create_all`):

```bash
alembic upgrade head        # применить все миграции
alembic revision --autogenerate -m "что изменилось"   # после правки моделей
alembic downgrade -1        # откатить последнюю
```

Если переносите УЖЕ существующую БД (созданную локально через `create_all`),
проштампуйте её, чтобы Alembic не пытался пересоздать таблицы:

```bash
alembic stamp head
```

## nginx / TLS

Пример конфига — `deploy/nginx.conf`. Важное: для `/api/events` (SSE) отключена
буферизация (`proxy_buffering off`), иначе события не доходят в реальном времени.
TLS терминируйте на nginx/облачном балансировщике.

## ⚠️ Масштабирование: пока ОДИН воркер

`SessionManager` и реестр SSE-подписчиков сейчас **in-process** (без Redis).
Поэтому:

- `gunicorn` запускается с `workers=1` (`WEB_CONCURRENCY=1`). Несколько воркеров =
  разные пулы сессий в разных процессах → события и сессии перестанут совпадать.
- Масштаб **вертикальный** (CPU/RAM одной машины). Узкое место — память браузеров
  (~0.5 ГБ + ~0.3 vCPU на активную сессию); регулируйте `MAX_SESSIONS` и
  `SESSION_IDLE_SECONDS`.
- Масштаб **вширь** (несколько воркеров/машин) требует ещё не реализованного слоя:
  Redis pub/sub для SSE + распределённая очередь задач + роутинг сессий по
  `user_id` (consistent hash) на выделенные воркеры. Это следующий шаг после
  первичного облачного запуска.

## Анти-бан и прокси

- Браузер по умолчанию headless, со stealth-настройками (UA/локаль/таймзона,
  маскировка `navigator.webdriver`), человекоподобными кликами/вводом и
  rate-лимитом (см. `hh_bot/antiban.py`).
- Прокси на пользователя задаётся в UI (хранится зашифрованно) либо глобальный
  фолбэк через `HH_PROXY_URL`. Для РФ-аккаунтов рекомендуются российские
  residential/mobile-прокси; пул подключается на стороне инфраструктуры.

## Данные и приватность

Хранятся PII и креды сторонних сайтов (зашифрованы). Размещайте БД и приложение
в регионе, допустимом для этих данных; том с профилями браузера держите на
зашифрованном диске.
